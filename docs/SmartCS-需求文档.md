## 一、项目概述

### 1.1 系统定位

SmartCS 是一套面向企业后勤报修/客服场景的智能工单系统。核心流程：

```
用户发起咨询 → 创建工单(已创建) → AI 搜索知识库回答
  ├─ AI 能回答 → 用户解决问题 → 超时自动关闭(已关闭)
  └─ AI 无法回答 → 转人工 → 客服处理(处理中) → 客服提请处理完成
       → 用户评价(已评价) → 客服关闭(已关闭)
```

### 1.2 技术栈

| 层 | 技术 | 版本 |
|---|---|---|
| 后端框架 | Flask (Python) | 3.8 |
| 数据库 | SQLite + WAL 模式 | 3.x |
| 前端 | Jinja2 模板 + 原生 JS | — |
| AI 引擎 | DeepSeek (OpenAI 兼容 API) | — |
| 反向代理 | Nginx | — |
| 进程管理 | systemd | — |

### 1.3 当前状态

- **app.py：** 3620 行，205 个函数
- **数据库表：** 27 张
- **API 端点：** 100+ 个 RESTful 接口
- **模板文件：** 6 个（chat.html, agent_dashboard.html, admin.html, agent_login.html, user_login.html, upload.html）
- **集成测试：** 62 个场景，全部通过 ✅
- **生产数据（当前）：** 42 张工单、59 个客户、293 条消息、81 条审计日志

### 3.3 工单状态机（重新设计）

**核心设计：仅注册用户可用，首次发消息即创建工单，工单贯穿全程**

```
注册用户登录
      │
      ▼
   ┌──────────┐     ╔═════════════════════════╗
   │  已创建    │ ◄──║ 已登录用户首次发消息    ║
   │ (created) │    ║ 即创建工单              ║
   └────┬─────┘    ║ 仅用户+管理员可见        ║
        │          ╚═════════════════════════╝
        │
   ┌────┼──────────────────────────────────────────────┐
   │    │ AI 回复，用户自解决                           │
   │    │                                              │
   │    └──► 用户不再回复 ═══▶ 超时自动关闭            │
   │                  (auto_close_min, 默认20分钟)      │
   │                                                   │
   └──── 用户点击转人工 ─────────────────────────────  │
                        │                             │
                        ▼                             │
                  ┌──────────┐                        │
                  │  处理中   │   客服受理，开始处理   │
                  │(processing)│                      │
                  └────┬─────┘                        │
                       │                              │
                客服回复、转交、图片                     │
                       │                              │
                       ▼                              │
                  ┌──────────┐                        │
                  │  已处理   │   客服点击             │
                  │ (resolved)│   「提请处理完成」     │
                  └────┬─────┘                        │
                       │                              │
           ┌───────────┼───────────┐                   │
           │                       │                   │
      用户确认+评价         超时未确认                  │
                       (auto_rate_hours, 默认24h)      │
           │                       │                   │
           ▼                       ▼                   │
        ┌─────────────────────────────┐                │
        │        已评价 (rated)        │               │
        │    等待客服/管理员关闭       │               │
        └──────┬──────────────────────┘                │
               │                                       │
    ┌──────────┼──────────┐                            │
    │                     │                            │
  客服关闭工单         管理员干预                        │
   (最后对接客服,        (强制处置关闭)                  │
    填写备注)                                           │
    │                     │                            │
    ▼                     ▼                            │
┌──────────────────────────────┐                       │
│        已关闭 (closed)         │◄─────────────────────┘
│        只读，不可回复          │
└──────────────────────────────┘
```

#### 状态定义

| 状态 | 英文常量 | 含义 | 触发条件 | 所有者 |
|------|---------|------|---------|--------|
| `已创建` | `created` | 工单建立，AI 自助阶段，等待用户操作 | 已登录用户首次发消息 | 用户/管理员（可见） |
| `处理中` | `processing` | 客服已受理，正在处理 | 客服点击受理 | 指定客服 |
| `已处理` | `resolved` | 客服处理完成，提请用户确认+评价 | 客服点击「提请处理完成」 | 用户 |
| `已评价` | `rated` | 用户已评价（或超时自动），等待客服/管理员关闭 | 用户提交评价或超时到达 | 最后对接客服 / 管理员 |
| `已关闭` | `closed` | 工单关闭，只读 | 客服或管理员关闭 | 只读 |

#### 流转规则

| 规则 | 说明 | 配置项（system_config） |
|------|------|------------------------|
| 用户前提 | **仅注册用户**可用，无游客场景 | — |
| 创建时机 | 已登录用户首次发消息即创建工单 | — |
| 可见性 | 已创建：仅**用户本人**和**管理员**可见，客服看不到 | — |
| AI 阶段 | 已创建状态下 AI 正常回复；用户可通过知识库自解决 | — |
| 超时自动关闭（已创建→关闭） | 已创建状态持续 N 分钟无用户消息，自动关闭 | `auto_close_min`（默认 20） |
| 转人工 | 用户点击转人工，工单进入处理中（客服可见） | — |
| 处理中→已处理 | 当前对接的客服点击「提请处理完成」 | — |
| 已处理→已评价 | 用户确认+评价（评分+反馈） | — |
| 已处理→已评价（超时） | 用户 N 小时内未确认，自动进入已评价 | `auto_rate_hours`（默认 24） |
| 已评价→已关闭 | 最后对接的客服填写备注，关闭工单 | — |
| 管理员干预 | 管理员可强制关闭任意状态的工单 | — |

#### 与现有系统的差异

| 差异点 | 当前实现（需改造） | 新设计 |
|--------|-----------------|-------|
| 用户门槛 | 游客可用（自动创建「访客xxx」） | **仅注册用户**，无游客 |

### 9.2 历史工单归档策略（已实现，需增强配置化）

**当前状态：** 已实现基础归档功能（`tickets_archive` 表 + 定时任务端点）

**设计目标：** 主表 `service_tickets` 保持轻量，历史数据自动迁移到 `tickets_archive` 并支持后续查询。

#### 归档流程

```
service_tickets（主表）
      │
      ├── 工单状态变为 closed
      │
      ├── 到达 N 天后
      │   （N = system_config.archive_after_days，默认 90）
      │
      ▼
  定时任务触发（每日凌晨）
      │
      ▼
  复制数据 → tickets_archive（归档表）
  删除主表记录（或保留摘要）
      │
      ▼
  用户/工程师仍可查询（API 自动合并主表和归档表）
```

#### 验收标准

| 功能 | 状态 | 说明 |
|------|------|------|
| `tickets_archive` 表 | ✅ 已存在 | 与 `service_tickets` 结构一致 + `archived_at` |
| 定时任务端点 | ✅ 已存在 | `/api/cron/archive` 每日触发 |
| 手动归档 API | ✅ 已存在 | `/api/admin/tickets/archive` |
| **归档天数可配置** | ⚠️ 需增强 | `system_config` 增加 `archive_after_days`（默认 90） |
| **归档后查询** | ⚠️ 需增强 | 用户端/工程师端/管理后台查询时自动合并主表和归档表结果 |
| **归档工单状态** | ✅ 已实现 | 仅 `closed` 状态的工单可归档 |
| 归档工单只读 | ✅ | archive 表数据不可修改 |

#### 归档数据查询策略

| 查询场景 | 查询范围 | 实现方式 |
|---------|---------|---------|
| 用户查看自己的工单 | 主表（活跃）+ 归档表（历史） | UNION ALL 合并 |
| 工程师查看经手工单 | 主表（活跃）+ 归档表（历史） | UNION ALL 合并 |
| 管理后台管理员查看 | 主表 + 归档表 | UNION ALL 合并 |
| 数据分析仪表盘 | 主表 + 归档表 | 分别统计后加总 |
| 按时间段搜索 | 主表 + 归档表 | 根据日期范围决定查询范围 |

> **注意：** 归档后如果数据量大（>100 万条），建议给 `tickets_archive` 加上 `created_at` 索引以保持查询速度。

| 旧状态数据迁移 | 现有工单需要做状态映射迁移 |
| 游客用户清理 | 无游客场景，`customer.name` 为「游客」的数据需处理 |

### 9.4 存储策略

**设计基准：** 每日 1,000 张工单

| 指标 | 10 年累计 | 说明 |
|------|----------|------|
| 总工单数 | ~3,650,000 张 | 含主表和归档表 |
| 总消息数 | ~55,000,000 条 | 约 10 亿字符 |
| 数据库文件 | ~16 GB | SQLite 单文件 |
| 图片/文件 | ~40 GB | 上传至本地磁盘 |
| 知识库 | ~2 GB | 文档 + FTS5 索引 |
| 可用磁盘（当前） | ~31 GB | ⚠️ 建议扩展至 100 GB

### 9.4 数据库迁移决策

- 当前：SQLite WAL（1,000 单/天舒适区）
- 未来：>5,000 单/天时迁移 PostgreSQL
- 迁移方案：环境变量 `DB_ENGINE` 切换 + pgloader（适配器模式）

| 指标 | 计算 | 数值 | 影响 |
|------|------|------|------|
| 工程师总数 | — | **80 人** | — |

| 总轮询请求 | 800 + 240 | **~1,040 次/分** | ✅ SQLite 承载 |
| 单请求耗时 | ≈1ms 缓存 / ≈50ms 查库 | — | ⚠️ 高峰期注意 |
| 数据库连接数 | Flask 单连接 | 1 | ✅ 无压力 |

**结论：** 80 人工程师规模，当前 Flask + SQLite 可承载。建议部署 **Gunicorn 3-4 workers** 避免单 worker 阻塞导致消息延迟。
